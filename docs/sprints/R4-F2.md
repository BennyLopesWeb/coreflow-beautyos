# Release 4 — Sprint R4-F2

**Desligar dual-write outbound (`project_*`) por padrão** (`2.5.0-r4-f2`).

| Campo | Valor |
|-------|-------|
| **Versão** | `2.5.0-r4-f2` |
| **Status** | ✅ Implemented (tech) |
| **Anterior** | R4-F1 ✅ |
| **RFC** | RFC-003 M7 · ADR-024 (sunset) · ADR-025 · ADR-027 · ADR-033 |

## Contexto

R4-F1 encerrou as rotas HTTP legado de booking (410 Gone). O último acoplamento
outbound que resta entre `core_bookings` e o legado é o dual-write (ADR-024):
os commands de booking (`create`/`approve`/`reject`/`cancel`) ainda chamam
`LegacyBookingAdapter.project_*`, criando/atualizando `agendamentos` +
`Payment` a cada operação. R4-F2 desliga esse dual-write **por padrão**,
mantendo o model `Agendamento` e as linhas históricas intactas — apenas para
de receber **novas** escritas quando a flag está OFF.

## Escopo IN

| # | Entrega |
|---|---------|
| 1 | Feature flag `FEATURE_BOOKING_LEGACY_PROJECTION_ENABLED` (`booking.legacy.projection.enabled`), default `false` |
| 2 | `CreateBookingHandler`: sem flag, não chama `project_create_booking`; booking core-only (`legacy_agendamento_id=None`, `mark_core_only_synced()`) |
| 3 | `ApproveBookingHandler` / `RejectBookingHandler` / `CancelBookingHandler`: removida a exigência de `legacy_agendamento_id`; `project_*` só é chamado com flag ON **e** id legado presente |
| 4 | `booking_reconciliation_worker.detect_drift`: `legacy_agendamento_id is None` deixa de ser drift — só há drift quando o id legado existe e está órfão/divergente |
| 5 | `QueueEntryService.aprovar_com_horario`: `entry.agendamento_id` fica `None` quando não há projeção (sem levantar erro) |
| 6 | `LegacySchedulingPortAdapter.check_availability`: passa a checar conflito diretamente em `core_bookings` (necessário porque bookings core-only não aparecem mais em `agendamentos` — preserva paridade P02/P09) |
| 7 | `PaymentReservationService.confirmar_deposito` sincroniza `CoreBooking.deposit_paid`; novo `confirmar_deposito_por_booking(booking_id)` para o path core-only (sem `Agendamento`) |
| 8 | `booking_port.py`: `project_*` documentados como opcionais (mantidos só para rollback) |
| 9 | Novos testes `test_r4_f2_dual_write_off.py`; testes de paridade legados (R2-F1/F2/F2b, CF8/CF9, staging D6) passam a ligar a flag de projeção explicitamente onde precisam do legado |

## Escopo OUT

| Item | Release |
|------|---------|
| Remover código `project_*` / `LegacyBookingAdapter` | R4-F3+ |
| Drop model/tabela `agendamentos` | R4-F3+ (requer zero uso da flag em produção) |
| Remover `Payment`/`Schedule` legado | R4-F3+ |

## Rollback

Reativar `FEATURE_BOOKING_LEGACY_PROJECTION_ENABLED=true` (env ou settings)
restaura o dual-write outbound imediatamente — nenhum deploy de código é
necessário. Bookings criados enquanto a flag estava OFF permanecem
core-only (sem projeção retroativa automática).
