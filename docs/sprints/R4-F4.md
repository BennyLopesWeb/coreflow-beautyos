# Release 4 — Sprint R4-F4

**Hard sunset de `Agendamento` — Option A (sem DROP TABLE)** (`2.7.0-r4-f4`).

| Campo | Valor |
|-------|-------|
| **Versão** | `2.7.0-r4-f4` |
| **Status** | ✅ Implemented (tech) |
| **Anterior** | R4-F3 ✅ |
| **RFC** | RFC-003 M8 (novo) · ADR-024 (sunset) · ADR-025 · ADR-027 · ADR-028 · ADR-033 |

## Contexto

R4-F3 removeu o código do dual-write outbound (`project_*`): todo booking
criado via `POST /v1/bookings` já era core-only, sem gerar linha em
`agendamentos`. Porém dois caminhos de escrita de produção continuavam
inserindo diretamente na tabela legado: `FilaService.aprovar_fila` (via
`AgendamentoService.criar_agendamento`) e, por extensão, qualquer chamada
administrativa que reutilizasse `criar_agendamento`. Além disso,
`DisponibilidadeService` e `QueueEntryService.processar_reservas_do_dia`
só liam `agendamentos` — não viam bookings core-only, o que já era
mascarado em criação (ACL `_has_core_booking_conflict`) mas quebrava a
correção da tela de disponibilidade e da fila operacional do dia.

R4-F4 fecha essas duas lacunas **sem remover a tabela física**
(Option A): nenhum caminho de escrita de produção pode mais inserir em
`agendamentos`; `core_bookings` passa a ser a fonte da verdade (SoT)
também para leitura de disponibilidade e processamento da fila do dia.
`agendamentos` permanece no banco somente para consulta histórica.

## Mudanças de código

- `app/core/config.py`: `APP_VERSION = "2.7.0-r4-f4"`.
- `infra/terraform/environments/{staging,prod}/terraform.tfvars.json`: tag `Version` → `2.7.0-r4-f4`.
- `app/services/fila_service.py`: `aprovar_fila` resolve catalog/offering via `LegacyCatalogAdapter` e delega a `CreateBookingHandler`; não chama mais `AgendamentoService.criar_agendamento`.
- `app/services/agendamento_service.py`: `criar_agendamento` levanta `BusinessRuleError` sempre; lógica de criação legado (validações, insert, payment pendente) removida (não há mais chamador); `_tem_conflito_intervalo` removido (código morto).
- `app/services/disponibilidade_service.py`: `_slots_ocupados` consulta `core_bookings` (novo `_duracao_core_booking`) além de `agendamentos`; união dos dois conjuntos.
- `app/services/queue_entry_service.py`: `processar_reservas_do_dia` dividido em `_processar_agendamentos_legado_do_dia` (histórico) + `_processar_core_bookings_do_dia` (primário, novo).
- `app/routers/admin.py`: novo endpoint `POST /admin/pagamentos/booking/{booking_id}/confirmar-sinal`.
- `app/models/agendamento.py`: docstring de depreciação no model `Agendamento`.
- `frontend/src/services/agendamentoService.ts`: `criar` sem fallback para rota legado (410).

## Débito residual

- **DROP físico de `agendamentos`/`payments`/`schedules`** — planejado para
  R4-F5, condicionado a:
  - Migrar/arquivar dados históricos necessários para auditoria/relatórios.
  - Confirmar que `OrderLegacySyncService`/`InvoiceLegacySyncService`/
    `PaymentLegacySyncService`/`booking_reconciliation_worker` não
    precisam mais desses dados (ou foram redirecionados/descontinuados).
  - Remover FKs físicas dependentes (`Fila.agendamento_id`,
    `QueueEntry.agendamento_id`, `Payment.agendamento_id`,
    `Financeiro.agendamento_id`, `CoreBooking.legacy_agendamento_id`).
- **Admin de pagamentos ainda dual-path** — `POST /admin/pagamentos/{agendamento_id}/confirmar-sinal`
  (legado) e `POST /admin/pagamentos/booking/{booking_id}/confirmar-sinal`
  (novo) coexistem. Reescrita completa do admin para booking-first fica
  para R4-F5, quando não houver mais reservas legado ativas.
- **Fila/QueueEntry sem FK para `core_bookings`** — o vínculo de bookings
  core-only na fila do dia usa dedupe por atributos compostos
  (`cliente_id`/`data`/`tranca_id`/`service_image_id`/`horario_entrada`)
  em vez de uma coluna nova (evitado para não exigir migração de schema
  nesta sprint). Se a heurística compor risco em produção, avaliar
  `core_booking_id` nullable em R4-F5.
- **`QueueEntryService.checkin`/`iniciar`/`concluir` não avançam
  `core_bookings`** — esses métodos só atualizam o status do booking
  legado quando `entry.agendamento_id` está preenchido; para entradas
  core-only (`agendamento_id is None`, já o caso desde
  `aprovar_com_horario` em R4-F3) o booking core não avança de status
  automaticamente durante o atendimento. Gap pré-existente desde R4-F3,
  não introduzido por esta sprint; correção completa (checkin/iniciar/
  concluir operarem sobre `CoreBooking`) fica para R4-F5.
