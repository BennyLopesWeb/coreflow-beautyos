# Release 4 — Sprint R4-F3

**Remover código do dual-write outbound (`project_*`)** (`2.6.0-r4-f3`).

| Campo | Valor |
|-------|-------|
| **Versão** | `2.6.0-r4-f3` |
| **Status** | ✅ Implemented (tech) |
| **Anterior** | R4-F2 ✅ |
| **RFC** | RFC-003 M7 (completo) · ADR-024 (sunset) · ADR-025 · ADR-027 · ADR-033 |

## Contexto

R4-F2 desligou o dual-write outbound (`project_*`) **por padrão**, mantendo o
código intacto atrás de `FEATURE_BOOKING_LEGACY_PROJECTION_ENABLED` como
kill-switch de rollback. Após período de observação sem uso da flag em
produção, R4-F3 remove **definitivamente** esse código: não há mais
possibilidade de reativar o dual-write via configuração — bookings são
sempre core-only. O modelo `Agendamento`/tabela `agendamentos` **não é
removido** nesta sprint (fora de escopo — ver Escopo OUT); permanece apenas
para dados históricos.

## Escopo IN

| # | Entrega |

| 1 | Removida `FEATURE_BOOKING_LEGACY_PROJECTION_ENABLED` de `config.py`, `feature_flags.py` (`booking.legacy.projection.enabled`) e `.env.example` |
| 2 | `CreateBookingHandler`/`ApproveBookingHandler`/`RejectBookingHandler`/`CancelBookingHandler`: removidos os branches condicionais de projeção — path é sempre core-only (`mark_core_only_synced()`, sem `project_*`) |

## Mudanças de código

- `app/core/config.py` / `app/core/feature_flags.py` / `.env.example`: flag removida.
- `app/modules/booking/application/commands/{create,approve,reject,cancel}_booking.py`: sem `feature_flags.is_enabled("booking.legacy.projection.enabled")`; sem `self.booking_port` (dependência da ACL removida dos handlers, já que só era usada para `project_*`/`resolve_legacy_ids`).
- `app/shared/acl/booking_port.py`: `LegacyBookingAdapter` reduzido a `create/approve/reject/cancel_booking_via_legacy` (fail-fast R3-F2) + `sync_booking_from_agendamento` (inbound).
- Docstrings de `scheduling_port.py`, `payment_reservation_service.py`, `booking_reconciliation_worker.py`, `queue_entry_service.py`, `reservation_service.py` atualizadas para refletir que o comportamento core-only agora é permanente (não mais condicional a flag).

## Rollback

Não há mais kill-switch de configuração — o dual-write outbound foi removido
do código. Rollback requer reverter o commit/PR (`git revert`) ou aplicar um
patch reintroduzindo `project_*`. O modelo `Agendamento` e dados históricos
permanecem intactos, então não há perda de dados associada a este rollback.

## Débito residual

- Tabela `agendamentos` (e `payments`/`schedules` legado associados)
  permanece no banco apenas para consulta histórica (`OrderLegacySyncService`,
  `InvoiceLegacySyncService`, `PaymentLegacySyncService`, reconciliation).
  Dropar essas tabelas é trabalho de **R4-F4 ou posterior**, condicionado a:
  - Confirmar que nenhum relatório/consulta financeira depende mais de
    leitura direta de `agendamentos`/`payments` legado.
  - Migrar ou arquivar dados históricos necessários para auditoria.
  - Remover `OrderLegacySyncService`/`InvoiceLegacySyncService`/
    `PaymentLegacySyncService` ou redirecioná-los para fontes core.

## Próximo

R4-F4 (ou posterior) — avaliar drop físico de `agendamentos`/`payments`
legado após confirmação de que nenhum consumidor ainda depende deles.
