# Release 4 — Sprint R4-F6

**Bridge `Payment`→`booking_id` + cutover disponibilidade core-only + 410 admin legado** (`2.9.0-r4-f6`).

## Contexto

R4-F5 fechou os débitos operacionais de `QueueEntry`/`Fila` (FK `booking_id`
+ avanço de status durante o atendimento) e marcou o admin de pagamentos
legado como `deprecated`. O gate R4-F5 documentou o bloqueio físico do DROP
de `agendamentos`/`payments`/`schedules`: **`Payment` e `Schedule` legado
ainda não tinham par core**, então removê-los exigiria antes migrar esses
dois domínios.

R4-F6 ataca esse bloqueio pelo lado do `Payment` (bridge para
`booking_id`) e pelo lado do `Schedule` (parar escrita em paths ativos),
além de completar a reescrita do admin de pagamentos para booking-first
(removendo — via 410 — a rota legado). **Não executa o DROP físico** —
isso continua fora de escopo, adiado explicitamente para **R4-F7**, porque
`Schedule` (model) e `SatisfactionSurvey` ainda referenciam `agendamentos`
por FK física, e a tabela `agendamentos` ainda é necessária para leitura
histórica (`DisponibilidadeService`, `AdminService.listar_pagamentos`,
sync services legado→core).

## Débito residual

- **DROP físico de `agendamentos`/`payments`/`schedules`** — continua
  planejado para **R4-F7**, condicionado a:
  - Remover FKs físicas dependentes restantes (`Schedule.agendamento_id`,
    `SatisfactionSurvey` — se aplicável —, `Fila.agendamento_id`,
    `QueueEntry.agendamento_id`, `Financeiro.agendamento_id`,
    `CoreBooking.legacy_agendamento_id`).
  - Migrar/arquivar dados históricos necessários para auditoria/relatórios.
  - Confirmar que não há mais consumidores lendo `agendamentos`/`payments`/
    `schedules` (sync services, reconciliation, relatórios, admin).
- **`AdminService.listar_pagamentos` ainda lista somente `Agendamento`
  legado** — bookings core-only não aparecem na tela admin de pagamentos
  (sem `booking_id` no schema/response); qualquer sinal pendente de um
  booking novo só é confirmável via `POST /admin/pagamentos/booking/
  {booking_id}/confirmar-sinal` chamado diretamente (não há botão na UI
  para isso ainda).
- **Reagendamento sem equivalente core** — `solicitar_reagendamento`/
  `aceitar_reagendamento` continuam restritos a `Agendamento` legado
  histórico (sem router HTTP ativo); não há endpoint `/v1/bookings`
  equivalente nesta release.
