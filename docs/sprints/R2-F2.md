## 2. Objetivo

Introduzir transições **`pending → approved`** e **`pending → rejected`** no aggregate `Booking` e `BookingDomainService`, com branch `booking.core.enabled` idêntico ao create (F1).

Com flag **ON** (ADR-024/026):

- Core SoT — transição de estado + incremento `version` (ADR-031 SM-3)  
- Legado = projeção outbound na mesma TX (ADR-025)  
- `PaymentQueryPort.is_deposit_confirmed()` antes de approve (ADR-028)  
- Eventos `booking.approved` / `booking.rejected` + aliases ADR-027  
- Resolver **TD-R2-F1b-001** — `OutboxService.defer_commit` para TX única  

**Resultado mensurável:** P03, P04, P05, P08 PASS flag ON e OFF; FF-BKG-001 ERROR; FF-PAY-001 ERROR.

## 7. Ordem de implementação (obrigatória)

Sequência para evitar refactor caro — **optimistic lock nasce com transição de estado**.

```
(0) OutboxService.defer_commit          ← TD-R2-F1b-001
        │
        v
(1) Aggregate: approve() / reject()     ← ADR-026 state machine
        │
        v
(2) BookingDomainService                ← PaymentQueryPort gate approve
        │
        v
(3) CoreBookingRepository               ← version increment + conditional UPDATE
        │
        v
(4) PaymentQueryPort + adapter          ← ADR-028 ACL
        │
        v
(5) Handlers branch flag ON/OFF         ← Approve + Reject
        │
        v
(6) ACL project_approve / project_reject
        │
        v
(7) Events + alias + correlation_id
        │
        v
(8) Router: ETag GET, If-Match POST
        │
        v
(9) Paridade P03–P05, P08
```

### State machine (subset R2)

```
                    create (F1)
                        │
                        v
                    ┌─────────┐
                    │ pending │
                    └────┬────┘
           approve()     │      reject()
         [deposit ok]    │      [always if pending]
                ┌────────┴────────┐
                v                 v
          ┌──────────┐      ┌──────────┐
          │ approved │      │ rejected │  ← terminal (SM-2)
          └──────────┘      └──────────┘
```

### Optimistic lock (ADR-031)

```sql
UPDATE core_bookings
SET status = :new_status,
    approved_at = :ts,          -- approve only
    version = version + 1,
    updated_at = :now
WHERE id = :id
  AND company_id = :cid
  AND version = :expected_version;
-- rows_affected = 0 → 409 version_conflict
```

## 8. Fluxo aprovado

### Approve — flag ON (core path)

```
POST /v1/bookings/{id}/approve
Headers: Idempotency-Key (opt), If-Match (opt), X-Correlation-Id
    ↓
IdempotencyStore.check (se key)
    ↓
Load aggregate via CoreBookingRepository
    ↓
PaymentQueryPort.is_deposit_confirmed(booking_id, company_id)
    ├─ false → 409 deposit_required (P04)
    └─ true  → continuar
    ↓
Booking.approve()  — valida pending (SM-1)
    ↓
BEGIN TRANSACTION (defer_commit outbox)
    ↓
CoreBookingRepository.save()     ← version N→N+1
    ↓
LegacyBookingPort.project_approve()
    ↓
Outbox: booking.approved + reservation.approved alias
    ↓
IdempotencyStore.save (se key)
    ↓
COMMIT
```

### Approve — flag OFF (ACL F0.5)

Comportamento atual via `LegacyBookingAdapter` — regressão idêntica; enriquecer com ETag/correlation apenas.

**Falha projeção → ROLLBACK total** (ADR-025).
