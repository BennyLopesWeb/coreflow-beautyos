# Release 4 — Sprint R4-F8

**DROP físico de `agendamentos`** (`2.11.0-r4-f8`).

| Campo | Valor |
|-------|-------|
| **Versão** | `2.11.0-r4-f8` |
| **Status** | ✅ Implemented (tech) |
| **Anterior** | R4-F7 ✅ |
| **RFC** | RFC-003 M11+ · ADR-024 (sunset) · ADR-025 · ADR-027 · ADR-028 · ADR-033 |

## Contexto

R4-F7 removeu a última FK física apontando para `agendamentos.id` (sete
tabelas), deixando o campo livre para o DROP. Esta sprint executa o DROP
físico propriamente dito e converte `Agendamento` de model SQLAlchemy
mapeado para um stub de compatibilidade (enums preservados, sem
`Base`/`__tablename__`) — encerrando o padrão Strangler Fig aberto desde
R2-F1 para o par `Agendamento`↔`CoreBooking`.

## Débito residual

- **`confirmar_deposito_por_booking` não popula `Financeiro`
  automaticamente** — bookings core-only não geram entrada financeira
  (`Financeiro`) ao confirmar o sinal, diferente do fluxo legado antigo
  (`AgendamentoService.confirmar_sinal` chamava
  `FinanceiroService.registrar_entrada_automatica`). Mitigado
  parcialmente: `FinanceiroService.registrar_entrada_automatica` já
  aceita `agendamento_id=None`, faltando só o call-site em
  `PaymentReservationService.confirmar_deposito_por_booking`.
- **`schedules`/`payments`/`financeiro`/`fila`/`queue_entries`/
  `notification_logs` mantêm a coluna `agendamento_id` como `Integer`
  histórico** (sem FK desde R4-F7) — não são candidatas a remoção de
  coluna nesta release; útil para auditoria de dados pré-migração.
- **Reagendamento e pagamento final sem equivalente core-only** —
  inalterado desde R4-F6/R4-F7 (`ReservationService.solicitar_reagendamento`/
  `aceitar_reagendamento`/`registrar_pagamento_final` são no-ops que
  sempre levantam `NotFoundError`; não há endpoint `/v1/bookings/...`
  equivalente ainda).
- **`POST /pagamentos/sinal` e `/pagamentos/sinal/gerar`/`/comprovante/{id}`
  legado** (router `app/routers/pagamentos.py`) sempre falham (400) —
  dependem de `AgendamentoService`, que agora sempre levanta
  `NotFoundError`. Não cobertos pelo middleware 410 (só rotas de booking
  o são). Aceito conscientemente: nenhum cliente ativo depende desses
  endpoints (substituídos por `/admin/pagamentos/booking/{id}/...` desde
  R4-F5/R4-F6).
