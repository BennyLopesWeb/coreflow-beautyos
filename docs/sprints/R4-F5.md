## Contexto

R4-F4 (hard sunset, Option A — sem DROP TABLE) garantiu que nenhum caminho
de escrita de produção insere mais em `agendamentos` e que `core_bookings`
passa a ser SoT para disponibilidade e processamento da fila do dia. O
gate R4-F4 documentou três débitos residuais explícitos:

1. `QueueEntryService.checkin`/`iniciar`/`concluir` só avançavam o status
   do booking quando `entry.agendamento_id` estava preenchido — para
   entradas core-only (o caso comum desde R4-F3/R4-F4, com
   `agendamento_id is None`) o `CoreBooking` nunca avançava de status
   durante o atendimento.
2. `QueueEntry`/`Fila` não tinham FK dedicada para `core_bookings` — o
   dedupe da fila do dia dependia de uma heurística por atributos
   compostos (`cliente_id`/`data`/`tranca_id`/`service_image_id`/
   `horario_entrada`), frágil em cenários de concorrência/edição manual.
3. Admin de pagamentos permanecia dual-path (`agendamento_id` legado +
   `booking_id` novo) sem nenhuma sinalização de depreciação no OpenAPI.
4. DROP físico de `agendamentos`/`payments`/`schedules` bloqueado por FKs
   de `Payment`/`Schedule` ainda legado — fora de escopo até esses dois
   models migrarem para o core.

R4-F5 fecha os débitos 1–3. O item 4 (DROP físico) permanece fora de
escopo — explicitamente adiado para **R4-F6**, condicionado à migração
de `Payment`/`Schedule` para o core.

## Débito residual

- **DROP físico de `agendamentos`/`payments`/`schedules`** — continua
  planejado para **R4-F6**, condicionado a:
  - Migrar `Payment`/`Schedule` legado para modelos core dedicados
    (hoje só `Agendamento`/`CoreBooking` têm par core; `Payment` e
    `Schedule` ainda não têm equivalente core — bloqueio físico do DROP).
  - Migrar/arquivar dados históricos necessários para auditoria/relatórios.
  - Remover FKs físicas dependentes (`Fila.agendamento_id`,
    `QueueEntry.agendamento_id`, `Payment.agendamento_id`,
    `Financeiro.agendamento_id`, `CoreBooking.legacy_agendamento_id`,
    `Schedule.agendamento_id`).
- **Admin de pagamentos ainda dual-path** — endpoint legado marcado
  `deprecated=True` nesta sprint, mas ainda funcional (reservas
  históricas ativas); reescrita completa booking-first (remoção da rota)
  fica para **R4-F6**.
- **Dedupe de `processar_reservas_do_dia` para entradas antigas sem
  `booking_id`** — bookings core criados antes desta migração (sem
  `booking_id` retroativo, coluna adicionada apenas para linhas novas)
  continuam usando o fallback por atributos compostos; sem backfill
  retroativo nesta sprint (fora de escopo — dados de produção existentes
  são poucos e de curto prazo, dado que R4-F4 é recente).
