# Release 4 — Sprint R4-F7

**Decouple físico das FKs restantes para `agendamentos`** (`2.10.0-r4-f7`).

| Campo | Valor |
|-------|-------|
| **Versão** | `2.10.0-r4-f7` |
| **Status** | ✅ Implemented (tech) |
| **Anterior** | R4-F6 ✅ |
| **RFC** | RFC-003 M11 (novo) · ADR-024 (sunset) · ADR-025 · ADR-027 · ADR-028 · ADR-033 |

## Contexto

R4-F6 fechou o bridge `Payment`→`booking_id` e o cutover de disponibilidade
core-only (com leitura de compatibilidade sobre `Agendamento` legado
mantida), além de marcar `ScheduleService.criar_para_reserva` como
deprecated. O gate R4-F6 documentou o bloqueio físico do DROP de
`agendamentos`/`payments`/`schedules`: sete tabelas ainda tinham FK física
para `agendamentos.id` — `payments`, `schedules` (`NOT NULL`),
`satisfaction_surveys` (`NOT NULL`), `fila`, `queue_entries`, `financeiro`
e `notification_logs`.

R4-F7 remove essa dependência física em todas as sete tabelas, mantendo a
tabela `agendamentos` (sem DROP — CF6/CF9 e sync legado→core ainda
precisam do model para fixtures/histórico) e completa o cutover de
disponibilidade core-only removendo a última leitura de compatibilidade
sobre `Agendamento` em `DisponibilidadeService`. **Não executa o DROP
físico** — isso continua fora de escopo, adiado explicitamente para
**R4-F8**.

## Débito residual

- **DROP físico de `agendamentos`/`payments`/`schedules`** — planejado
  para **R4-F8**, condicionado a:
  - Migrar/arquivar dados históricos necessários para auditoria/relatórios.
  - Confirmar que não há mais consumidores lendo `agendamentos`/`payments`/
    `schedules` (sync services, reconciliation, relatórios, admin).
- **`AdminService.listar_pagamentos` ainda lista somente `Agendamento`
  legado** — bookings core-only não aparecem na tela admin de pagamentos
  (débito herdado de R4-F6, inalterado nesta release).
- **Disponibilidade não bloqueia mais slot por `Agendamento` histórico
  ativo** — aceito conscientemente: nenhum caminho de escrita ativo cria
  `Agendamento` novo desde R4-F3/R4-F4, então o risco prático é limitado a
  dados legado pré-migração ainda pendentes (mitigado por
  `_expirar_agendamentos_pendentes`, que continua ativo).
- **Reagendamento sem equivalente core** — inalterado desde R4-F6.
