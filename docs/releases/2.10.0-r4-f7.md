# Release Notes — CoreFlow BeautyOS `2.10.0-r4-f7`

**Data:** 2026-07-21
**Codename:** R4-F7 Decouple FKs — remove físico das FKs restantes para `agendamentos`
**Branch:** `r4-f7-decouple-fks-drop-prep`

## Destaques

- **`agendamentos` deixa de ser referenciada por FK física em qualquer
  tabela.** A migration `cf015_r4_f7_decouple_agendamento_fks` remove a
  constraint `FOREIGN KEY ... REFERENCES agendamentos(id)` de sete
  tabelas — `payments`, `schedules`, `satisfaction_surveys`, `fila`,
  `queue_entries`, `financeiro` e `notification_logs`. As colunas
  `agendamento_id` permanecem como `Integer` simples em todas, preservadas
  apenas para leitura histórica.
- **`Schedule`/`SatisfactionSurvey` ganham `booking_id`.** Mesmo padrão
  adotado em `Payment` no R4-F6: `agendamento_id` passa a ser nullable e
  `booking_id` (FK nullable `core_bookings.id`, indexada) é adicionado —
  fecha o último bloqueio físico documentado no gate R4-F6 para o DROP.
- **Disponibilidade 100% core-only.** `DisponibilidadeService
  ._slots_ocupados` remove a última leitura de compatibilidade sobre
  `Agendamento` legado (mantida desde R4-F4/R4-F6) — `core_bookings` é a
  única fonte de ocupação da agenda a partir desta release.
- **`agendamentos` continua existindo.** Não é DROP — a tabela permanece
  necessária para fixtures (CF6/CF9) e leitura histórica (sync
  legado→core, relatórios).

## Sem DROP TABLE (continua Option A)

Esta release **não remove** `agendamentos`/`payments`/`schedules`. O DROP
físico continua **explicitamente adiado para R4-F8**, condicionado a:

- Migrar/arquivar dados históricos necessários para auditoria/relatórios.
- Confirmar que não há mais consumidores lendo `agendamentos`/`payments`/
  `schedules` (sync services, reconciliation, relatórios,
  `AdminService.listar_pagamentos`).

## Sem breaking changes de API

- Nenhum endpoint HTTP foi alterado nesta release — mudanças são
  exclusivamente de schema físico (FK) e de uma leitura interna
  (`DisponibilidadeService`).
- `Schedule`/`SatisfactionSurvey`/`Payment`/`Fila`/`QueueEntry`/
  `Financeiro`/`NotificationLog` — aditivo: `agendamento_id` sem
  constraint física, `booking_id` novo em `Schedule`/`SatisfactionSurvey`.
  Nenhum campo removido.
- **Mudança de comportamento intencional (não-API):**
  `DisponibilidadeService` deixa de considerar `Agendamento` legado
  histórico ativo como ocupante de slot — apenas `core_bookings`. Como
  nenhum caminho de escrita ativo cria `Agendamento` novo desde
  R4-F3/R4-F4, o impacto prático fica limitado a dados legado
  pré-migração ainda pendentes (mitigado por
  `expirar_reservas_pendentes`).

## Débito residual (R4-F8)

- DROP físico de `agendamentos`/`payments`/`schedules` legado.
- `AdminService.listar_pagamentos` ainda lista somente `Agendamento`
  legado — bookings core-only não aparecem na tela admin de pagamentos.
- Reagendamento (`solicitar_reagendamento`/`aceitar_reagendamento`) sem
  endpoint core equivalente — continua restrito a `Agendamento` legado
  histórico, sem router HTTP ativo.
