# Release Notes — CoreFlow BeautyOS `2.7.0-r4-f4`

**Data:** 2026-07-21
**Codename:** R4-F4 Hard Sunset Agendamento Writes (Option A, sem DROP)
**Branch:** `r4-f4-sunset-agendamentos-model`

## Destaques

- **Nenhum caminho de escrita de produção insere mais em `agendamentos`.**
  `FilaService.aprovar_fila` migrado para `CreateBookingHandler` (mesmo
  padrão de `QueueEntryService.aprovar_com_horario` desde R4-F3);
  `AgendamentoService.criar_agendamento` desativado — levanta
  `BusinessRuleError` apontando para `POST /v1/bookings`.
- **`core_bookings` passa a ser SoT também para leitura de disponibilidade
  e fila do dia.** `DisponibilidadeService._slots_ocupados` une ocupação
  de `core_bookings` ativos (primário) com `agendamentos` históricos
  (leitura, dados anteriores à migração).
  `QueueEntryService.processar_reservas_do_dia` processa `core_bookings`
  `APPROVED` do dia além do histórico legado.
- **Novo endpoint admin** `POST /admin/pagamentos/booking/{booking_id}/confirmar-sinal`
  — aceita `booking_id` direto (via `PaymentReservationService.confirmar_deposito_por_booking`),
  sem exigir `Agendamento`. O endpoint legado por `agendamento_id`
  permanece para reservas históricas.
- Model `Agendamento` marcado com docstring `.. deprecated:: 2.7.0-r4-f4`
  — DROP físico planejado para **R4-F5**.
- Frontend: `agendamentoService.criar` remove o fallback morto para
  `POST /agenda/agendamentos` (rota já retornava `410 Gone` desde R4-F1) —
  usa exclusivamente `POST /v1/bookings`.

## Sem DROP TABLE (Option A)

Esta release **não remove** a tabela `agendamentos` nem seus dados. O
model é mantido **somente leitura** para histórico
(`listar_agendamentos`/`buscar_por_id`/`obter_agendamento`,
sync services legado→core, relatórios, reconciliation). O DROP físico
fica para R4-F5, condicionado à migração/arquivamento de dados
históricos e à remoção das FKs físicas dependentes
(`Fila.agendamento_id`, `QueueEntry.agendamento_id`,
`Payment.agendamento_id`, `Financeiro.agendamento_id`,
`CoreBooking.legacy_agendamento_id`).

## Sem breaking changes de API

- Rotas de leitura (`GET /agenda/agendamentos`, `GET /agenda/agendamentos/{id}`)
  continuam funcionando (somente leitura, sem alteração).
- `POST /admin/pagamentos/{agendamento_id}/confirmar-sinal` (legado)
  continua funcionando para reservas históricas existentes.
- Bookings criados em releases anteriores com `legacy_agendamento_id`
  preenchido (dual-write ativo em R2/R3, ou janela R4-F2) continuam
  sendo lidos/processados normalmente.

## Débito residual (R4-F5+)

- DROP físico de `agendamentos`/`payments`/`schedules` legado e FKs
  associadas.
- Reescrita completa do admin de pagamentos para booking-first (hoje
  coexistem dois endpoints — legado por `agendamento_id` e novo por
  `booking_id`).
- `QueueEntry`/`Fila` sem FK dedicada para `core_bookings` — dedupe da
  fila do dia usa atributos compostos em vez de coluna nova (evitado
  para não exigir migração de schema nesta sprint).
- `QueueEntryService.checkin`/`iniciar`/`concluir` não avançam o status
  de `core_bookings` quando a entrada é core-only (gap pré-existente
  desde R4-F3, documentado aqui para acompanhamento).
