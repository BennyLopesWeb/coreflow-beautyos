# Release Notes — CoreFlow BeautyOS `2.11.0-r4-f8`

**Data:** 2026-07-22
**Codename:** R4-F8 Drop Agendamentos — DROP físico da tabela `agendamentos`
**Branch:** `r4-f8-drop-agendamentos`

## Destaques

- **`agendamentos` é removida fisicamente.** A migration
  `cf016_r4_f8_drop_agendamentos` executa `DROP TABLE agendamentos`
  (idempotente — só se a tabela existir) em produção; `migrate_schema.py`
  espelha o DROP para o caminho SQLite legado. Nenhuma tabela mais
  referencia `agendamentos` desde R4-F7 (FK física já havia sido
  removida), então o DROP não tem nenhum bloqueio físico.
- **`Agendamento` deixa de ser um model SQLAlchemy mapeado.** A classe
  permanece no código-fonte só para referência/import (enums
  `ReservationStatus`/`StatusPagamento`/`StatusAgendamento`/
  `STATUS_OCUPAM_VAGA` continuam em uso ativo por `CoreBooking` e
  colunas históricas), mas não herda mais `Base` — `create_all` nunca
  mais recria a tabela, e `session.query(Agendamento)` levanta erro
  imediatamente (classe não mapeada).
- **Toda leitura/escrita de produção sobre `Agendamento` foi reescrita.**
  Services que antes consultavam a tabela (`AgendamentoService`,
  `ReservationService`, `PaymentReservationService`,
  `DisponibilidadeService`, `ScheduleService`, `QueueEntryService`,
  `NotificationService`, sync services legado→core,
  `booking_reconciliation_worker`, `LegacyPaymentQueryAdapter`) agora
  são no-ops seguros (retornam vazio/`None` ou levantam `NotFoundError`)
  ou consultam `CoreBooking` equivalente (`AgenteService`,
  `AgendaDiaService`, `AdminService`).

## Sem breaking changes de API core

- Nenhum endpoint `/v1/*` (CoreFlow) foi alterado nesta release —
  `POST /v1/bookings`, `GET /v1/payments?booking_id=`,
  `POST /admin/pagamentos/booking/{id}/confirmar-sinal` continuam
  funcionando exatamente como antes.
- **Mudança de comportamento intencional em rotas legado não-booking:**
  `POST /pagamentos/sinal`, `/pagamentos/sinal/gerar` e
  `/pagamentos/comprovante/{agendamento_id}` (router
  `app/routers/pagamentos.py`) sempre retornam `400` agora — dependem de
  `AgendamentoService`, que sempre levanta `NotFoundError` (tabela
  removida). Essas rotas já não têm clientes ativos desde R4-F5/R4-F6
  (substituídas por `/admin/pagamentos/booking/{id}/...`); não estão
  cobertas pelo middleware `410 Gone` (que só cobre rotas de booking).

## Sem impacto em dados core

- `core_bookings`, `core_payments`, `core_orders`, `core_invoices` e
  demais tabelas `core_*` não são afetadas — o DROP é exclusivo da
  tabela legado `agendamentos`.
- `payments`/`schedules`/`fila`/`queue_entries`/`financeiro`/
  `notification_logs` mantêm a coluna `agendamento_id` (Integer, sem FK
  desde R4-F7) — preservada apenas para auditoria histórica, sem
  qualquer join/consulta ativa contra `agendamentos`.

## Débito residual (para releases futuras)

- `PaymentReservationService.confirmar_deposito_por_booking` ainda não
  cria uma entrada `Financeiro` automaticamente para bookings
  core-only (gap identificado; `FinanceiroService.registrar_entrada_automatica`
  já aceita `agendamento_id=None`, falta só o call-site).
- Reagendamento (`solicitar_reagendamento`/`aceitar_reagendamento`) e
  pagamento final (`registrar_pagamento_final`) sem endpoint core-only
  equivalente — inalterado desde R4-F6/R4-F7.
- Rotas legado de `app/routers/pagamentos.py` (`/pagamentos/sinal` e
  correlatas) continuam existindo, mas sempre falham (400) — candidatas
  a sunset explícito (410) em release futura.
