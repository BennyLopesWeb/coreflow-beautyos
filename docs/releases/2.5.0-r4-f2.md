# Release Notes — CoreFlow BeautyOS `2.5.0-r4-f2`

**Data:** 2026-07-21
**Codename:** R4-F2 Disable Booking Legacy Dual-Write Projection
**Branch:** `r4-f2-remove-booking-dual-write`

## Destaques

- Nova feature flag `FEATURE_BOOKING_LEGACY_PROJECTION_ENABLED`
  (`booking.legacy.projection.enabled`), **default `false`** — dual-write
  outbound (`project_create_booking`/`project_approve_booking`/
  `project_reject_booking`/`project_cancel_booking`) deixa de ser chamado
  por padrão pelos commands de booking.
- Bookings criados com a flag OFF são **core-only**: `legacy_agendamento_id`
  permanece `None` e nenhuma linha nova é criada em `agendamentos`/`payments`.
- Approve/reject/cancel funcionam normalmente em bookings core-only (a
  exigência de `legacy_agendamento_id` foi removida dos commands).
- `booking_reconciliation_worker`: `legacy_agendamento_id is None` deixa de
  ser tratado como drift — só há drift quando existe id legado órfão ou
  status divergente.
- `LegacySchedulingPortAdapter` passa a checar conflitos diretamente em
  `core_bookings`, preservando a proteção contra double-booking mesmo sem
  projeção legado.
- Novo `PaymentReservationService.confirmar_deposito_por_booking` confirma
  sinal diretamente no `CoreBooking` (sem exigir `Agendamento`).

## Sem breaking changes

- Model `Agendamento` e dados históricos **permanecem intactos**.
- Rotas físicas de booking legado **não são removidas** além do que já foi
  feito em R4-F1 (410 Gone).
- `project_*` continuam implementados em `LegacyBookingAdapter` — apenas
  deixam de ser chamados por padrão. Reativar a flag restaura o
  comportamento anterior sem deploy de código (kill-switch de rollback).

## Rollback

```bash
export FEATURE_BOOKING_LEGACY_PROJECTION_ENABLED=true
```
