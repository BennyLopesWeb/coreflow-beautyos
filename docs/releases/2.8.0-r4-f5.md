# Release Notes — CoreFlow BeautyOS `2.8.0-r4-f5`

**Data:** 2026-07-21
**Codename:** R4-F5 Core Ops — FK booking_id + fechamento do gap operacional
**Branch:** `r4-f5-drop-agendamentos-fks`

## Destaques

- **`QueueEntry`/`Fila` ganham vínculo forte com `core_bookings`.** Nova
  coluna `booking_id` (FK nullable, indexada) substitui a heurística por
  atributos compostos usada até R4-F4 para dedupe da fila do dia.
  `aprovar_com_horario`, `processar_reservas_do_dia` e
  `FilaService.aprovar_fila` passam a preencher `booking_id` sempre que
  criam/linkam um booking core.
- **`checkin`/`iniciar`/`concluir` avançam `CoreBooking.status` para
  entradas core-only.** Fecha o gap operacional documentado no gate
  R4-F4: antes, esses métodos só atualizavam status quando a entrada
  tinha `agendamento_id` (legado) — bookings core-only nunca avançavam
  de status durante o atendimento. Entradas legado puras continuam com o
  fallback `Agendamento`/`ReservationService`.
- **Admin de pagamentos legado marcado como deprecated.**
  `POST /admin/pagamentos/{agendamento_id}/confirmar-sinal` aparece como
  `deprecated` no OpenAPI e tem docstring apontando para o path
  booking-first (`POST /admin/pagamentos/booking/{booking_id}/confirmar-sinal`).
  Endpoint ainda funcional — reescrita completa fica para R4-F6.

## Sem DROP TABLE (continua Option A)

Esta release **não remove** `agendamentos`/`payments`/`schedules`. O
DROP físico planejado anteriormente para "R4-F5" foi **explicitamente
adiado para R4-F6**, porque:

- `Payment` e `Schedule` legado ainda não têm modelo core equivalente —
  removê-los exigiria antes migrar esses dois domínios para o core
  (fora do escopo desta sprint, focada em `QueueEntry`/`Fila`).
- FKs físicas dependentes (`Fila.agendamento_id`,
  `QueueEntry.agendamento_id`, `Payment.agendamento_id`,
  `Financeiro.agendamento_id`, `CoreBooking.legacy_agendamento_id`,
  `Schedule.agendamento_id`) continuam apontando para `agendamentos`.

## Sem breaking changes de API

- `QueueEntryResponse`/`FilaResponse`/`FilaItemDetalhado` ganham novo
  campo opcional `booking_id` — aditivo, sem remoção de campos
  existentes.
- Endpoint legado `POST /admin/pagamentos/{agendamento_id}/confirmar-sinal`
  continua funcionando (apenas marcado `deprecated` no schema OpenAPI).
- Bookings criados antes desta release (sem `booking_id` retroativo —
  sem backfill) continuam sendo processados normalmente via fallback por
  atributos compostos em `processar_reservas_do_dia`.

## Débito residual (R4-F6+)

- DROP físico de `agendamentos`/`payments`/`schedules` legado e FKs
  associadas — condicionado à migração de `Payment`/`Schedule` para o
  core.
- Reescrita completa do admin de pagamentos para booking-first (remoção
  da rota legado, hoje apenas marcada deprecated).
- Sem backfill retroativo de `booking_id` para `QueueEntry`/`Fila`
  criadas antes desta release — dedupe por atributos compostos mantido
  como fallback para esses registros antigos.
