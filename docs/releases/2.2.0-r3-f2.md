# Release Notes — CoreFlow BeautyOS `2.2.0-r3-f2`

**Data:** 2026-07-16
**Codename:** R3-F2 Remove ReservationService Booking Delegation
**Branch:** `r3-f2-remove-reservation-booking`

## Destaques

- Booking write path passa a ser **core-only**: `ReservationService` (criar/aprovar/rejeitar/cancelar) e a ACL `*_via_legacy` não delegam mais escrita ao legado — sempre levantam `BusinessRuleError`.
- `FEATURE_BOOKING_CORE_ENABLED` agora é **`true` por padrão**. Desligar a flag não restaura o legado — funciona como **kill-switch de emergência** (bloqueia escrita com HTTP 409).
- Eventos alias `reservation.*` (ADR-027) **deixam de ser publicados** pelos commands de booking — apenas `booking.*` no outbox. As factories alias permanecem no catálogo (para R4) e emitem `logger.warning` se chamadas.
- Dual-write `project_*` (Agendamento legado, pagamentos, schedule) **preservado** — leituras, pagamentos, reagendamento e conclusão continuam funcionando normalmente.
- Routers `/reservations` e `/agendamentos` **não foram removidos** (previsto para R3-F3) — mas a escrita subjacente via `ReservationService` agora falha (HTTP 400) apontando para `/v1/bookings`.

## Feature flags (defaults)

| Flag | Default |
|------|---------|
| `FEATURE_BOOKING_CORE_ENABLED` | **true** (era `false`) |
| `FEATURE_RESOURCE_ENGINE_ENABLED` | false |
| `FEATURE_PLUGIN_ENGINE_ENABLED` | false |
| `CORE_ENFORCEMENT_MODE` | off (local) / block (staging & production) |

## Breaking changes

- `POST/PUT/DELETE /reservations/*` e `POST/PUT/DELETE /agendamentos` (legado) agora retornam erro (400/409) em qualquer ambiente, independentemente do `CORE_ENFORCEMENT_MODE` — a causa raiz passou a ser a remoção do write path em `ReservationService`, não apenas o enforcement HTTP.
- Consumidores de eventos que ainda dependem de `reservation.created`, `reservation.approved`, `reservation.rejected`, `reservation.cancelled` **deixam de receber esses eventos** a partir desta versão. Migre para `booking.created`, `booking.approved`, `booking.rejected`, `booking.cancelled` (ADR-027).
- `FEATURE_BOOKING_CORE_ENABLED=false` deixa de ser um modo operacional válido para escrita — é apenas kill-switch (409 em toda tentativa de escrita de booking).

## Migração

Clientes ainda usando `/reservations` ou `/agendamentos` para criar/aprovar/rejeitar/cancelar devem migrar para:

| Legado | Successor |
|--------|-----------|
| `POST /reservations`, `POST /agendamentos` | `POST /v1/bookings` |
| `PUT /reservations/{id}/approve` | `POST /v1/bookings/{id}/approve` |
| `PUT /reservations/{id}/reject` | `POST /v1/bookings/{id}/reject` |
| `DELETE /reservations/{id}` | `POST /v1/bookings/{id}/cancel` |

Eventos: assine `booking.*` em vez de `reservation.*` (superset de payload — nenhum campo removido).

## Rollback

```bash
export FEATURE_BOOKING_CORE_ENABLED=false
```

⚠️ Isso **bloqueia** toda escrita de booking (HTTP 409) — não restaura o comportamento legado. Para reverter o código desta sprint, use `git revert <merge-commit-r3-f2>`.
