# Release Notes — CoreFlow BeautyOS `2.6.0-r4-f3`

**Data:** 2026-07-21
**Codename:** R4-F3 Remove Booking Legacy Dual-Write Code
**Branch:** `r4-f3-remove-project-dual-write`

## Destaques

- Removida **definitivamente** a feature flag
  `FEATURE_BOOKING_LEGACY_PROJECTION_ENABLED` (`booking.legacy.projection.enabled`)
  de `config.py`, `feature_flags.py` e `.env.example`.
- Removidos do código `LegacyBookingAdapter`/`BookingPort`
  (`app/shared/acl/booking_port.py`): `project_create_booking`,
  `project_approve_booking`, `project_reject_booking`,
  `project_cancel_booking`, `resolve_legacy_ids` e `get_booking_legacy_id`
  (sem uso restante fora do dual-write removido).
- `CreateBookingHandler`/`ApproveBookingHandler`/`RejectBookingHandler`/
  `CancelBookingHandler` não têm mais nenhum branch condicional de
  projeção — todo booking é **sempre core-only**
  (`legacy_agendamento_id=None`).
- `sync_booking_from_agendamento` (sincronização **inbound** legado → core)
  e os métodos `*_via_legacy` (fail-fast, R3-F2) permanecem inalterados.

## Sem breaking changes

- Model `Agendamento` e dados históricos **permanecem intactos** — apenas o
  código de escrita outbound (`project_*`) foi removido, não o modelo nem a
  tabela.
- Rotas físicas de booking legado continuam retornando `410 Gone` (R4-F1) —
  nenhuma mudança de comportamento HTTP.
- Bookings criados em releases anteriores com projeção legado (dual-write
  ainda ativo em R2/R3, ou flag reativada durante a janela R4-F2) continuam
  funcionando normalmente — a leitura/reconciliação de
  `legacy_agendamento_id` existente não foi alterada.

## Sem kill-switch de rollback

Diferente de R4-F2, esta release **remove o código**, não apenas desliga um
comportamento por flag. Não é mais possível reativar o dual-write outbound
via variável de ambiente. Rollback requer `git revert` do commit/PR.

## Débito residual (R4-F4+)

- Tabela `agendamentos` (e `payments`/`schedules` legado associados)
  permanece no banco apenas para dados históricos e para os serviços de
  sync legado→core (`OrderLegacySyncService`, `InvoiceLegacySyncService`,
  `PaymentLegacySyncService`, `booking_reconciliation_worker`). O drop
  físico dessas tabelas é trabalho de uma release futura (R4-F4 ou
  posterior).
