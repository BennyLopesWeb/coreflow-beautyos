# Release Notes — CoreFlow BeautyOS `2.9.0-r4-f6`

**Data:** 2026-07-21
**Codename:** R4-F6 Payment Bridge — booking_id + disponibilidade core-only + 410 admin legado
**Branch:** `r4-f6-core-payments-schedule-drop-prep`

## Destaques

- **`Payment` ganha vínculo com `core_bookings` sem depender de
  `Agendamento`.** `payments.agendamento_id` passa a ser opcional e
  `payments.booking_id` (FK nullable, indexada) é adicionado —
  `PaymentReservationService.criar_pendente` aceita bookings core-only, e
  `confirmar_deposito_por_booking` agora cria/atualiza (best-effort) uma
  linha em `payments` vinculada por `booking_id`, dando paridade de
  auditoria com o path legado.
- **Disponibilidade explicitamente core-only para bookings novos.**
  `DisponibilidadeService._slots_ocupados` documenta `core_bookings` como
  única fonte de ocupação para reservas novas; a união com `Agendamento`
  legado é mantida só como leitura de compatibilidade histórica.
  `expirar_reservas_pendentes` agora também expira `CoreBooking` pendente
  sem sinal pago (via `CancelBookingHandler`), sem depender de
  criar/cancelar `Agendamento`.
- **Parada de escrita em `Schedule`.** `ReservationService
  .aceitar_reagendamento` — o único caminho de escrita de `Schedule`
  restante — deixou de chamar `ScheduleService.criar_para_reserva`. O
  model `Schedule` **não foi removido**.
- **Admin de pagamentos legado removido (410 Gone).**
  `POST /admin/pagamentos/{agendamento_id}/confirmar-sinal` responde
  sempre `410 Gone` com `successor` apontando para
  `POST /admin/pagamentos/booking/{booking_id}/confirmar-sinal` (path
  único desde esta release). A rota permanece registrada no OpenAPI
  (`deprecated=True`) só para discoverability.

## Sem DROP TABLE (continua Option A)

Esta release **não remove** `agendamentos`/`payments`/`schedules`. O DROP
físico continua **explicitamente adiado para R4-F7**, porque:

- `Schedule` (model) e `SatisfactionSurvey` ainda têm FK física para
  `agendamentos.id`.
- `agendamentos` ainda é necessária para leitura histórica
  (`DisponibilidadeService`, `AdminService.listar_pagamentos`, sync
  services legado→core).
- FKs físicas dependentes (`Fila.agendamento_id`,
  `QueueEntry.agendamento_id`, `Payment.agendamento_id`,
  `Financeiro.agendamento_id`, `CoreBooking.legacy_agendamento_id`,
  `Schedule.agendamento_id`, `SatisfactionSurvey.agendamento_id`)
  continuam apontando para `agendamentos`.

## Sem breaking changes de API

- `Payment`/`payments` — aditivo: `agendamento_id` relaxado para
  opcional, `booking_id` novo (nullable). Nenhum campo removido.
- `POST /admin/pagamentos/booking/{booking_id}/confirmar-sinal`
  inalterado.
- **Breaking change intencional:** `POST /admin/pagamentos/
  {agendamento_id}/confirmar-sinal` deixa de retornar `200` e passa a
  responder sempre `410 Gone` (comportamento já sinalizado como
  `deprecated` desde `2.8.0-r4-f5`). Clientes que ainda dependem dessa
  rota devem migrar para o path booking-first.

## Débito residual (R4-F7+)

- DROP físico de `agendamentos`/`payments`/`schedules` legado e FKs
  associadas — condicionado à remoção das últimas FKs físicas
  (`Schedule`/`SatisfactionSurvey`) e confirmação de que não há mais
  consumidores lendo dados legado.
- `AdminService.listar_pagamentos` ainda lista somente `Agendamento`
  legado — bookings core-only não aparecem na tela admin de pagamentos.
- Reagendamento (`solicitar_reagendamento`/`aceitar_reagendamento`) sem
  endpoint core equivalente — continua restrito a `Agendamento` legado
  histórico, sem router HTTP ativo.
